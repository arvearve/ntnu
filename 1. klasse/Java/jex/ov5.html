Overview

This exercise focuses on encapsulation: How to hide the internal representation of an object to the outside world with getters, and to prevent an object to be in an illegal state through setters.

  3. Person class with additional encapsulation methods. (20 pt.)

Part 2: Person class properly implementing and encapsulating a bi-directional parent/child association (25 pt.)

In this part, we focus on the bi-directional parent/child association.
public class Person in package oving5.person (25 pt.)
The Person class propertly implements and encapsulates a bi-directional parent/children association. The parent/child association relates a Person to a father and a mother (both Persons), in one direction, and a Person to zero, one or more children (also Persons), in the other direction. The mother and father must be encapsulated with getFather()/setFather(Person) and getMother()/setMother(Person) methods, respectively, while the children must be encapsulated getChildCount(), getChild(int), addChild(Person) and removeChild(Person) methods. Note that it is illegal to set the father of a Person to a female or the mother to a male.

Special care must be taken to make sure that the parent/child association is kept consistent:
  • If a child has a parent, either the mother or father, the parent must also have that child as one of its children, and vica versa.
  • A child cannot have a parent, unless that parent also has that child as one of its children.
Hence, setParent, addChild and removeChild must all take care to call each other (or helper methods that you introduce), as appropriate.

Making sure the parent/child association is consistent is more tricky than you may think. It may be helpful to draw an object diagram for the various cases you must handle, before coding.

Part 3: Person class with additional encapsulation methods. (20 pt.)

In this part, we implement additional utility methods in the Person class.
public class Person in package oving5.person (20 pt.)
The Person class must implement additional encapsulation methods.

public int method indexOfChild in public class Person in package oving5.person
This method that takes a Person as argument and returns the index of that Person among the children of this Person. If the Person argument is not a child of this Person, the method should return -1.

This method is the opposite of getChild: If person.getChild(i) == child then person.indexOfChild(child) == i.
public boolean method containsChild in public class Person in package oving5.person
Method that returns true if the Person argument is a child of this Person, and false otherwise.

public boolean method isAncestorOf in public class Person in package oving5.person
Returns true if this Person is an ancestor of the provided Person argument, and false otherwise. A Person is an ancestor of another Person, if
  • this Person is the father or mother of the other Person or
  • this Person is the ancestor of the other Person's father or mother.

    • One way of doing it would be to create a helper method that takes two Person objects as arguments and check whether or not one is the ancestor of the other. Use this helper method to check if this is an ancestor of the fathers father and mothers mother and so on, until a person is null or the ancestor is the father or mother. This means that the helper method must call itself recursively.

    Part 4: Card and CardDeck-classes (40 pt.)

    In this part, we implement the Card and CardDeck classes, with appropriate getters, setters and constructors.
    public class Card in package oving5.card (10 pt.)
    The Card class will be very similar to the one in a previous exercise, but the properties suit and face will be encapsulated. You have have to implement getter methods to retrieve their values. By not implementing setter methods you ensure these properties cannot be changed after a Card is created.

    public class CardDeck in package oving5.card (30 pt.)
    The CardDeck class will be very similar to the one in a previous exercise, but the list of Cards will be encapsulated and a deal(int) method will be added.

    public constructor CardDeck in public class CardDeck in package oving5.card
    The constructor should take no arguments, and initialize this CardDeck with the cards in the correct order (suits S, H, D, C, faces 1 to 13).

    public int method getCardCount in public class CardDeck in package oving5.card
    This method should return the number of cards in this CardDeck.

    Note that the card count may be lower than 52 because of the deal method.
    public Card method getCard in public class CardDeck in package oving5.card
    This method should return the Card at the specified index among this CardDeck's cards. If the index is not between 0 and the card count, null should be returned.

    public ArrayList<Card> method deal in public class CardDeck in package oving5.card
    Takes an int argument and returns a list with that many Cards removed from the top of this CardDeck. Just as a real deck, the cards are dealt one by one from the top (highest index) of this CardDeck, and each card should only be dealt once - meaning you have to keep track of how many cards have been dealt. If there are not enough cards left in the this CardDeck, the method should return null.

    The order is important: The first card removed from (the top of) this CardDeck, should be first in the returned list, the second card removed should be second in the list, and so forth. I.e. if deal(4) is called just after the CardDeck is created, it should return a list containing the King of Clubs (since Ace has face value 1 and not 14), Queen of Clubs, Knight of Clubs and Ten of Clubs, in that order.
